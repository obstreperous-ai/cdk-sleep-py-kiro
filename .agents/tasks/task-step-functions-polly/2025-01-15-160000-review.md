# Step Functions state machine skeleton with Polly integration

The change introduces an AWS Step Functions Standard Workflow (`AudioPipelineStateMachine`) containing a single `CallAwsService` task that invokes `polly:startSpeechSynthesisTask` with hardcoded placeholder parameters. The EventBridge rule, previously targeting a CloudWatch Log Group, now targets the state machine with `$.detail` as input. The log group is retained as the state machine's logging destination. Six new tests in a dedicated file verify the state machine, its Polly task, logging, IAM, EventBridge wiring, and input forwarding.

**Watch for:** The state machine execution role bundles Polly and CloudWatch Logs permissions into a single `Resource: "*"` statement (confirmed). The Polly task will fail at runtime because it references a non-existent `"placeholder-bucket"` and no error handling exists in the definition (confirmed — intentional skeleton, but the missing catcher means EventBridge retries will hammer a permanently-failing execution).

---

## High-level view

The state machine is a linear two-state chain (PollyTask → Succeed) wired to EventBridge via `SfnStateMachine` target with `InputPath: $.detail`. CDK's `CallAwsService` generates the `arn:{partition}:states:::aws-sdk:polly:startSpeechSynthesisTask` resource URI and auto-grants `polly:startSpeechSynthesisTask` on `*`. The result is a skeleton that synthesizes and passes tests but will fail every execution at runtime until the placeholder parameters are replaced and error handling is added.

The IAM posture is broader than necessary: CDK merges the logging permissions (8 actions on `*`) and the Polly permission into a single statement. Narrowing it requires a custom policy override — acceptable for a skeleton but worth addressing before production.

The test suite covers the structural contract well but does not assert any negative constraints — there's no test that the old CloudWatch Log Group target is absent from the rule, so a regression that adds both targets would go undetected.

1. **Polly + Logs permissions merged on `Resource: "*"`** — CDK's default policy merges `polly:startSpeechSynthesisTask` and 8 CloudWatch Logs delivery actions into one `Allow` statement on `*`. 3. **Placeholder bucket means guaranteed runtime failure** — `OutputS3BucketName: "placeholder-bucket"` will cause every execution to fail with an InvalidS3Bucket error. Acceptable as a skeleton marker, but the combination with no error handling (above) makes this a noisy failure mode. (confirmed)
4. **No negative test for removed Log Group target** — The old `CloudWatchLogGroup` target was removed from the rule, but no test asserts its absence. A future regression could add both targets silently. (confirmed)

## IAM policy surface

The synthesized `AudioPipelineStateMachineRoleDefaultPolicy` contains a single statement:

```json
{
  "Action": [
    "logs:CreateLogDelivery",
    "logs:DeleteLogDelivery",
    "logs:DescribeLogGroups",
    "logs:DescribeResourcePolicies",
    "logs:GetLogDelivery",
    "logs:ListLogDeliveries",
    "logs:PutResourcePolicy",
    "logs:UpdateLogDelivery",
    "polly:startSpeechSynthesisTask"
  ],
  "Effect": "Allow",
  "Resource": "*"
}
```

The 8 `logs:*` actions are required by Step Functions for CloudWatch Logs delivery and CDK generates them automatically. CDK merges these with the Polly permission into one statement because both use `Resource: "*"` — `iam_resources=["*"]` was specified on the `CallAwsService` construct. This means the Polly permission inherits the `*` resource from the Logs requirement with no way to scope it to specific resources without splitting the policy manually via a custom override.

## Failure mode without error handling

The state machine definition has no `Retry` or `Catch` on the PollyTask state. When the task fails (which it will, given the placeholder bucket), the execution transitions directly to a terminal failure. EventBridge rule targets have a built-in retry policy (up to 185 retries over 24 hours by default for Step Functions targets). Each retry starts a new execution that immediately fails, generating significant CloudWatch Logs volume at the ALL logging level.

The `SfnStateMachine` CDK construct supports `dead_letter_queue` and `retry_attempts` parameters that could limit this behavior. Neither is configured. For a skeleton that won't be triggered in production yet this is tolerable, but it becomes a problem the moment the input bucket receives real uploads in a deployed environment.

## EventBridge input forwarding

The target uses `InputPath: "$.detail"` which strips the outer EventBridge envelope (source, account, region, time, detail-type) and passes only the S3 event detail. Future states that need event metadata (e.g., timestamp for idempotency, account for cross-account routing) would require changing to an InputTransformer that selectively includes envelope fields alongside the detail.

## Test coverage gaps

Not tested:
- No assertion that the EventBridge rule does NOT have a CloudWatch Log Group target (only positive assertions exist) — a regression could silently add both targets
- The `iam_resources=["*"]` scope is not tested — the test confirms the action string but not the resource breadth
- No test for the state machine definition structure beyond string matching (the Succeed state, result_path, and parameters are not verified)
- The log group used for state machine logging is not pinned — the test accepts `any_value()` for the ARN

The existing `test_eventbridge_rule_has_target` in `test_cdk_base_stack.py` still passes but provides no additional signal now that the more specific Step Functions tests exist. Consider removing or marking it as superseded.
