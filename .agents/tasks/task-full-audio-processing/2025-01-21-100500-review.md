# Full audio processing pipeline with S3, Polly, and DynamoDB (v2)

The Lambda handler implements dual-mode processing: audio files are downloaded from the input bucket and re-uploaded to the output bucket, while text files are read from S3, synthesized to speech via Polly, and uploaded as MP3. The CDK stack grants S3 read/write and Polly IAM permissions, bumps Lambda resources to 512 MB / 60s, and the Step Functions `UpdateStatusCompleted` task now writes output metadata (outputKey, outputBucket, fileSize) from the Lambda response. The five issues from the v1 review have been addressed: Polly output is buffered into BytesIO before upload (fixing fileSize), the redundant Lambda COMPLETED write is removed, the unused `INPUT_BUCKET` env var is removed from CDK, a real-temp-file test validates audio fileSize, and temp cleanup is wrapped in try/finally.

**Watch for:** ARCHITECTURE.md still documents `INPUT_BUCKET` as a Lambda environment variable and references the removed `_update_dynamodb_success` function — the docs are out of sync with the code (confirmed). Polly's `synthesize_speech` API has a 3000-character limit for plain text input; no validation or chunking is performed, so text files exceeding that limit will throw `TextLengthExceededException` at runtime (confirmed). The `except (ClientError, Exception)` clause is redundant since `Exception` is a superclass of `ClientError` (confirmed, cosmetic).

**Verdict**: APPROVED

## High-level view

Step Functions is now the sole owner of the COMPLETED status transition, writing output metadata from `$.processAudioResult.Payload`. The Lambda only writes FAILED on error paths — a clean ownership split.

The remaining documentation drift in ARCHITECTURE.md will mislead future contributors trying to understand status ownership. The Polly character limit is an operational boundary that will surface as an unhandled exception for any text file over ~3000 characters — acceptable if the product only ingests short prompts, but worth documenting or guarding if longer content is expected.

## Polly text length boundary

The `_process_text_file` function reads the entire S3 object into memory and passes it directly to `polly_client.synthesize_speech(Text=text_content, ...)`. The Polly `SynthesizeSpeech` API accepts a maximum of 3000 characters for plain text (6000 for SSML). If a user uploads a text file exceeding that limit, the Lambda will receive a `TextLengthExceededException`:

```python
text_content = response["Body"].read().decode("utf-8")
# No length check here
polly_response = polly_client.synthesize_speech(
    OutputFormat="mp3",
    Text=text_content,  # Could be > 3000 chars
    VoiceId="Joanna",
)
```

The exception will be caught by the generic handler, DynamoDB will be marked FAILED, and Step Functions will route to the failure path. The error message won't tell the user *why* their file was rejected. A pre-check like `if len(text_content) > 3000: raise ValueError(...)` would surface a clear message. Alternatively, `start_speech_synthesis_task` supports up to 200,000 characters but requires an S3 output location (already available).

## ARCHITECTURE.md documentation drift

Three specific inaccuracies in the current ARCHITECTURE.md:

Line 234 lists `INPUT_BUCKET` as an environment variable. The CDK stack only sets `TABLE_NAME` and `OUTPUT_BUCKET`.

Line 317 states: "COMPLETED - Written by the Lambda directly (via `_update_dynamodb_success`) after successful processing, and also by `UpdateStatusCompleted`...". The Lambda no longer writes COMPLETED — that function was removed. Only Step Functions writes COMPLETED.

Line 375 in the "Happy Path" section states: "The Lambda updates DynamoDB directly with `status=COMPLETED`..." — also stale.

These discrepancies will mislead contributors trying to understand the status ownership model (Step Functions owns COMPLETED, Lambda owns FAILED).
