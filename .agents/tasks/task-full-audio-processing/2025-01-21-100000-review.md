# Full audio processing pipeline: S3, Polly text-to-speech, and DynamoDB metadata

The Lambda handler (`sleep_audio_processor/handler.py`) is extended from a validation-only stub into a full processing engine. It now downloads audio from the input S3 bucket (or synthesizes speech via Polly for `.txt` files), uploads results to the output bucket, and writes completion metadata to DynamoDB. The CDK stack adds the necessary grants (S3 read/write, `polly:SynthesizeSpeech`), environment variables (`INPUT_BUCKET`, `OUTPUT_BUCKET`), and bumps the Lambda to 512 MB / 60s timeout. Thirty-three new tests cover the Lambda logic and infrastructure assertions. Existing tests are adapted with `autouse` mock fixtures to prevent the now-real boto3 calls from escaping.

**Watch for:** The file-size calculation for Polly-generated audio reads the stream position *after* it has already been consumed by `upload_fileobj`, which will always return 0 (confirmed). The Lambda writes `COMPLETED` to DynamoDB, then Step Functions *also* writes `COMPLETED` via `UpdateStatusCompleted` — a redundant double-write that could race or mask partial failures (confirmed). The `INPUT_BUCKET` env var is set but never read for routing — the handler trusts the event's `bucket.name` field instead (confirmed).

## High-level view

The Polly audio stream is uploaded via `upload_fileobj` which consumes the stream; the subsequent `seek(0, 2)` / `tell()` to measure file size will always see position 0 since botocore streaming bodies aren't seekable after consumption. Every successful invocation reports `fileSize: 0` for text files, and that zero propagates to DynamoDB and the Step Functions response.

The Lambda and Step Functions both write COMPLETED status to DynamoDB for the same audioId. The Lambda writes immediately after upload; UpdateStatusCompleted writes again when it executes next. If the Lambda partially succeeds (upload done, DynamoDB update fails), the exception handler marks the record FAILED, then Step Functions also catches and enters the failure path — the record ends up FAILED even though the output file exists in S3.

Existing test files (`test_lambda_handler.py`, `test_pipeline_validation.py`, `test_error_handling_observability.py`) add `autouse` mock fixtures that patch the module-level boto3 clients. Any new test module that imports the handler must remember to patch all three clients or risk real AWS calls in CI.

<summary>Issues (5)</summary>

1. **Polly fileSize always zero** — `_process_text_file` calls `audio_stream.seek(0, 2)` after `upload_fileobj` has consumed the stream. Botocore `StreamingBody` is not seekable; even if it were, the upload has already drained it. Buffer the stream content before upload, or measure the byte length of what was uploaded.
2. **Redundant COMPLETED writes to DynamoDB** — The Lambda writes `status=COMPLETED` in `_update_dynamodb_success`, then the Step Functions `UpdateStatusCompleted` task writes `status=COMPLETED` again. The second write is idempotent but wastes a DynamoDB WCU on every invocation and masks the question of which component owns the "official" completion status.
3. **INPUT_BUCKET env var unused** — The handler never reads `INPUT_BUCKET`; it trusts `event["bucket"]["name"]` directly. Either add a validation check against the env var, or remove the env var as dead configuration.
4. **Audio file size depends on temp file existence** — `_process_audio_file` calls `os.path.getsize(tmp_path) if os.path.exists(tmp_path) else 0`. Since `download_file` is mocked in tests, no file is created, so `fileSize` is always 0 in test assertions — the tests don't actually verify size correctness.
5. **No temp-dir cleanup on exception** — If `upload_file` raises after `download_file` succeeds, the temp file in `/tmp` is never cleaned up. Over many retries or cold-start reuse, `/tmp` (512 MB in Lambda) could fill. Wrap the download-upload in a try/finally for cleanup.

## Polly stream consumption and file-size measurement

In `_process_text_file`, the handler uploads the Polly `AudioStream` with `s3_client.upload_fileobj(audio_stream, ...)`, then immediately attempts to measure the size:

```python
audio_stream.seek(0, 2)  # Seek to end
file_size = audio_stream.tell()
```

Botocore's `StreamingBody` does not support `seek`. The tests pass because the mock uses a `BytesIO` object (which *is* seekable), but in production the `AudioStream` is a `botocore.response.StreamingBody`. Calling `seek` on it raises `AttributeError` or returns 0. Either way the production path is broken for `.txt` inputs.

Fix: read the stream into a `BytesIO` buffer, measure `len(buffer.getvalue())`, then pass the buffer to `upload_fileobj`.

## Dual COMPLETED writes and ownership ambiguity

The Lambda calls `_update_dynamodb_success` which sets `status=COMPLETED, outputKey, outputBucket, fileSize, updatedAt`. Then Step Functions proceeds to the `PollyTask` placeholder, then `UpdateStatusCompleted`, which writes `status=COMPLETED, outputKey, outputBucket, fileSize, updatedAt` *again* — with different timestamp sources (Python `datetime.now` vs Step Functions `$$.State.EnteredTime`). The second write wins, overwriting `updatedAt` with the Step Functions clock.

If the Lambda's DynamoDB write is authoritative, the Step Functions task is wasted work. If Step Functions is authoritative, the Lambda shouldn't be writing status at all — just returning the payload. Pick one owner.

## Temp file lifecycle in the audio path

`_process_audio_file` downloads to `/tmp/{base_name}_{unique_id}{ext}`, measures size, uploads, then cleans up. The cleanup is in a bare `try/except OSError: pass` block *after* the upload. If the upload throws, execution jumps to the caller's `except` block and the temp file is never removed. Each Lambda container has 512 MB of `/tmp` space; repeated failures with large audio files could exhaust it before the container is recycled.

## Test coverage gaps

Not tested: the actual `fileSize` value under realistic conditions (tests always get 0 because mocked `download_file` creates no real file and mocked `BytesIO` seek behavior diverges from production `StreamingBody`). Not tested: behavior when `OUTPUT_BUCKET` env var is empty (handler would upload to bucket name `""`). Not tested: text files larger than Polly's 3000-character `synthesize_speech` limit (production would get a Polly `TextLengthExceededException`).
