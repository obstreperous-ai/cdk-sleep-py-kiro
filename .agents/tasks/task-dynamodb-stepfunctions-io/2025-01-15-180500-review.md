# DynamoDB metadata tracking for the audio pipeline state machine (v2)

The change adds a DynamoDB table (`SleepAudioMetadataTable`) and wires it into the Step Functions state machine as a metadata store. The pipeline now writes a PROCESSING record before invoking Polly, updates to COMPLETED on success, and catches errors on both WriteInitialRecord and PollyTask to route to a Fail state (with a FAILED record written only when possible). Table configuration is production-appropriate: on-demand billing, SSE, point-in-time recovery. Fourteen CDK assertion tests verify the table properties, state machine wiring, and error handling.

**Watch for:**

- `$.object.key` is used directly as `audioId`. If the S3 key contains path separators or URL-encoded characters (e.g., `folder/my%20file.mp3`), the DynamoDB item key will be the raw key string, which may produce surprising lookup behavior downstream. (likely)
- The chain-ordering test (`test_state_machine_chain_ordering`) relies on positional ordering of state names in the serialized ASL JSON, which is not guaranteed by the ASL specification—state names can appear in any order in the `States` object. A future CDK version could reorder them and break the test without changing behavior. (possible)

**Verdict**: APPROVED

## High-level view

All three issues from the v1 review have been addressed. WriteInitialRecord now has a catch routing to the Fail state, ARCHITECTURE.md accurately says "CDK-managed write access," and three new tests cover catch configuration, chain ordering, and the Fail state.

The remaining design-level observation is the `audioId` format contract: the raw S3 key is used without normalization, documented in a code comment and ARCHITECTURE.md, but the schema table doesn't specify that this is the raw unnormalized key including path prefixes—a downstream developer might expect a filename-only value.

The new chain-ordering test works today but relies on JSON key serialization order, which is an implementation detail of CDK's Python bindings rather than a guarantee of the ASL specification.

## Chain-ordering test fragility

The new `test_state_machine_chain_ordering` test does this:

```python
write_pos = definition_text.index("WriteInitialRecord")
polly_pos = definition_text.index("PollyTask")
update_completed_pos = definition_text.index("UpdateStatusCompleted")
assert write_pos < polly_pos
assert polly_pos < update_completed_pos
```

ASL definitions are JSON objects with a `States` key whose value is an unordered map. CDK currently serializes states in definition order, making this test pass, but the JSON specification does not guarantee key ordering. A more robust approach would parse the definition JSON and walk the `Next` field from `StartAt` to verify the chain. This isn't blocking—the test works today and will likely continue working with CDK's Python bindings—but it's a maintenance risk if someone investigates a spurious test failure after a dependency upgrade.

## audioId schema documentation gap

The ARCHITECTURE.md schema table says `audioId` is "Partition key - derived from the S3 object key" without specifying that it's the raw, unnormalized key including any path prefix (e.g., `uploads/2024/my%20file.mp3`). A downstream developer building a query interface might expect a filename-only value or a UUID. The code comment in `cdk_base_stack.py` explains this clearly, but the architecture doc's schema table is where someone building a consumer would look first.
