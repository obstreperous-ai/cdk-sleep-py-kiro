# DynamoDB metadata tracking for the audio pipeline state machine

The change adds a DynamoDB table (`SleepAudioMetadataTable`) and wires it into the Step Functions state machine as a metadata store. The pipeline now writes a PROCESSING record before invoking Polly, updates to COMPLETED on success, and catches errors to mark FAILED before terminating. Table configuration is production-appropriate: on-demand billing, SSE, point-in-time recovery. Eleven new CDK assertion tests verify the table properties and state machine definition content.

**Watch for:**

- The error-handling catch is scoped only to the PollyTask. A failure in WriteInitialRecord itself (e.g., DynamoDB throttle, malformed key) propagates uncaught and the execution terminates with no metadata record—leaving no trace that processing was attempted. (confirmed)
- `$.object.key` is used directly as `audioId`. If the S3 key contains path separators or URL-encoded characters (e.g., `folder/my%20file.mp3`), the DynamoDB item key will be the raw key string, which may produce surprising lookup behavior downstream. (likely)

## High-level view

The state machine definition grows from a two-state skeleton (PollyTask → Done) into a five-state pipeline with error branching. The Catch clause on the Polly task ensures the table always reflects terminal state on the Polly path, but the initial DynamoDB write itself is unguarded—if it fails, the execution errors out with no FAILED record written.

CDK auto-grants (`DynamoPutItem`/`DynamoUpdateItem` L2 constructs) produce an IAM policy broader than what the code strictly requires (includes `GetItem`, `DeleteItem`, `BatchWriteItem`, etc.). The ARCHITECTURE.md claims least-privilege scoped to PutItem/UpdateItem, which doesn't match the actual synthesized policy.

The test file uses string-matching against the serialized ASL definition—adequate for asserting presence of tasks and statuses, but doesn't verify wiring (catch configuration, chain ordering, result paths).

## WriteInitialRecord has no error catch

The `add_catch` is attached only to `polly_task`:

```python
polly_task.add_catch(update_status_failed, result_path="$.error")
```

If `WriteInitialRecord` fails—because the input event is missing `$.object.key` (a `States.Runtime` error) or the table is temporarily unavailable—the execution transitions directly to the Step Functions built-in error state. No FAILED record is written, and there's no observability beyond the execution history. The EventBridge rule fires on any `Object Created` event, so if the event shape deviates (schema version change, non-S3 trigger source), the initial write will fail with no trace in DynamoDB.

## audioId key format contract

The PutItem uses `$.object.key` directly as the partition key:

```python
"audioId": sfn_tasks.DynamoAttributeValue.from_string(
    sfn.JsonPath.string_at("$.object.key")
),
```

S3 object keys can be multi-segment paths (`audio/uploads/file.mp3`) and may contain URL-encoded characters in the EventBridge event detail. Any downstream service querying by `audioId` must know it's the raw S3 key, not a UUID or normalized identifier. The ARCHITECTURE.md says "derived from the S3 object key" but could be more explicit about the format contract.

## CDK-granted permissions vs. documented intent

CDK's L2 DynamoDB task constructs call `table.grantWriteData()` internally, which grants `PutItem`, `UpdateItem`, `GetItem`, `DeleteItem`, `DescribeTable`, `ConditionCheckItem`, `BatchGetItem`, `BatchWriteItem`, and `Query`/`Scan` on the table. The ARCHITECTURE.md states the role has permissions for `dynamodb:PutItem`/`UpdateItem`/`GetItem`/`DeleteItem`—this is closer but still incomplete. The documentation should either enumerate the actual grants or simply say "CDK-managed write access."

## Test coverage gaps

Not tested:

These are all verifiable via CDK assertions on the definition structure. The current tests prove the pieces exist but don't prove they're wired correctly.
