# SNS notification topics for pipeline completion and failure

Two KMS-encrypted SNS topics (Completed, Failed) are wired into the Step Functions state machine as terminal notifications — one on the happy path after status update, one on the error path after failure recording. A dedicated KMS CMK with rotation encrypts both topics. Eight new CDK assertion tests cover resource existence, encryption, IAM grants, state machine integration ordering, and error-handling preservation.

**Watch for:** The KMS key has no grant to the state machine role or the SNS service principal (confirmed). At runtime, `sns:Publish` to these encrypted topics will likely fail with a KMS access denied because the state machine role lacks `kms:GenerateDataKey*` and `kms:Decrypt` permissions. Additionally, neither SNS Publish step has error handling — if the notification itself fails, the execution hangs or fails without reaching the terminal state cleanly.

## High-level view

The notification steps slot into the existing chain with correct ordering: metadata is written before notifications fire, and `result_path="$.snsResult"` avoids clobbering the execution state.

CDK's `SnsPublish` task calls `topic.grant_publish()` internally, which grants `sns:Publish` but does not grant KMS permissions on the encryption key. The synthesized template confirms: the state machine role's policy has `sns:Publish` but zero KMS actions. This is a deployment-time success but a runtime failure.

Neither notification step has an `add_catch`. On the success path, if SNS publish fails, the entire execution fails — making a successfully-processed file look like a pipeline failure. On the error path, the execution enters an unhandled-error terminal rather than the explicit `Fail` state.

<details>
<summary>Issues (3)</summary>

1. **Missing KMS permissions for encrypted publish** — The state machine role needs `kms:GenerateDataKey*` and `kms:Decrypt` on the SNS encryption key. Add `sns_kms_key.grant_encrypt_decrypt(state_machine.role)` after the state machine is created, or use `sns_kms_key.grant(state_machine.role, 'kms:GenerateDataKey*', 'kms:Decrypt')`. (confirmed)
2. **No error handling on notification steps** — `PublishCompletedNotification` and `PublishFailedNotification` have no `add_catch`. A transient SNS/KMS failure on the success path causes the execution to fail without reaching `Done`, making a successful pipeline run look like a failure. Consider adding catch blocks that route to the terminal states (Done/Fail) so notification failure doesn't mask pipeline outcome. (confirmed)
3. **`$.snsResult` collision on error path** — Both SNS tasks write to `$.snsResult`. On the error path this is harmless (only one runs), but if future refactoring adds retries or parallel notifications, the shared result path becomes a data-clobbering risk. (possible)

## KMS encryption without publish-path grant

The synthesized CloudFormation template shows the state machine role's IAM policy contains three statement blocks: DynamoDB (`PutItem`, `UpdateItem`), a wildcard statement for logs and Polly, and `sns:Publish` scoped to the two topic ARNs. No KMS actions appear anywhere in the role's policy.

The KMS key policy grants `kms:*` to the account root principal, which enables IAM-policy-based access — but only if the IAM policy actually includes KMS actions. Since it doesn't, the state machine cannot use the key.

```
# Synthesized policy (abbreviated):
{ "Action": "sns:Publish", "Resource": [completed_topic, failed_topic] }
# Missing:
# { "Action": ["kms:GenerateDataKey*", "kms:Decrypt"], "Resource": key_arn }
```

The fix is a single line after the state machine is constructed:

```python
sns_kms_key.grant_encrypt_decrypt(state_machine.role)
```

CDK's `sfn_tasks.SnsPublish` does not do this automatically when the topic has a custom CMK — `grant_publish` handles the SNS permission but not the KMS grant.

## Notification steps lack error isolation

`PublishCompletedNotification` sits between `UpdateStatusCompleted` and `Done`. If SNS publish fails (network timeout, KMS error, throttle), the state machine execution fails entirely — even though the actual audio processing succeeded and the DynamoDB record is already marked COMPLETED. This means a transient notification failure makes the pipeline report failure for a file that was successfully processed.

On the error path: `PublishFailedNotification` sits between `UpdateStatusFailed` and `Fail`. If the notification itself fails, the execution enters an unhandled-error terminal rather than the explicit `Fail` state with its cause annotation. Less impactful (the execution still fails) but produces a different error surface in CloudWatch.

A catch block routing to the same terminal state would make notification failures non-blocking:

```python
publish_completed_notification.add_catch(succeed_state, result_path="$.notificationError")
publish_failed_notification.add_catch(fail_state, result_path="$.notificationError")
```

## Test coverage gaps

Not tested: KMS permissions on the state machine role (which is precisely the gap that exists), and behavior when the notification step itself fails. A test asserting `kms:GenerateDataKey*` in the policy would have caught the missing grant.

The ordering test uses string position comparison on the serialized definition — fragile if CDK changes serialization order, but adequate for the current state.
