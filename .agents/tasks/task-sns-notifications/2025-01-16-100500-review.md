# SNS notification topics for pipeline completion and failure (v2)

Two KMS-encrypted SNS topics (Completed, Failed) wired into the Step Functions state machine as terminal notifications. This revision addresses all three issues from the v1 review: KMS permissions are granted via `grant_encrypt_decrypt` on the state machine role, both SNS Publish steps have `add_catch` blocks routing to their respective terminal states, and result paths are now distinct (`$.snsCompletedResult`, `$.snsFailedResult`). Four new tests directly verify these fixes.

**Watch for:** ARCHITECTURE.md still documents `$.snsResult` as the shared result path (confirmed) â€” the code uses distinct paths but the docs don't match.

**Verdict**: APPROVED

## High-level view

The KMS permission gap is closed. `sns_kms_key.grant_encrypt_decrypt(state_machine.role)` is called after the state machine is constructed, producing IAM policy statements for `kms:Decrypt`, `kms:Encrypt`, `kms:ReEncrypt*`, and `kms:GenerateDataKey*`.

Error isolation is in place on both paths. A notification failure on the success path still reaches `Done` â€” the pipeline outcome is no longer masked by a transient SNS error. The error path catches notification failures and routes to `Fail` with the original cause preserved.

The result-path collision is resolved: `$.snsCompletedResult` and `$.snsFailedResult` are now distinct, enforced by a regex-based assertion test.

## Documentation drift on result paths

ARCHITECTURE.md (line 200) states:

> Both tasks store their result in `$.snsResult` to avoid overwriting the main execution state

The actual implementation:
```python
publish_completed_notification = sfn_tasks.SnsPublish(..., result_path="$.snsCompletedResult")
publish_failed_notification = sfn_tasks.SnsPublish(..., result_path="$.snsFailedResult")
```

This was the v1 issue #3 fix, but ARCHITECTURE.md wasn't updated to reflect the change. Documentation only, no runtime impact.

## Error-path catch semantics

On the failure path, `publish_failed_notification` has both `.next(fail_state)` and `.add_catch(fail_state, result_path="$.notificationError")`. Whether notification succeeds or fails, the execution ends at `Fail` â€” catch routes the same place as the happy-next. The `$.notificationError` field provides observability into whether the notification itself failed without altering the pipeline outcome.
