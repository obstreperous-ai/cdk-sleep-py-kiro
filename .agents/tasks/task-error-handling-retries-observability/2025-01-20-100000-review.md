# Retry policies, error routing, and observability for the audio pipeline

The change hardens the Step Functions pipeline with exponential-backoff retries on WriteInitialRecord, ProcessAudio, and PollyTask, narrows error catches from catch-all to specific error types, enables X-Ray tracing on both Lambda and the state machine, adds CloudWatch Alarms for execution failures and Lambda errors, and converts the Lambda handler to structured JSON logging with request ID correlation.

Watch for: The ProcessAudio catch block enumerates specific Lambda errors but omits the handler's own `ValueError` (and any other application-level exception), meaning a validation failure that raises before returning a result will fall through uncaught to a Step Functions execution failure rather than routing to UpdateStatusFailed (confirmed). The `PublishFailedNotification` message references `$.processAudioResult.Payload.validationError`, which will not exist when the error path is entered via a Catch (the state will have `$.error` populated instead), causing the notification task itself to fail silently (confirmed, pre-existing but newly relevant because the error routing is more specific now).

## High-level view

Retry policies follow a standard pattern: DynamoDB and Polly tasks retry on `States.TaskFailed` while the Lambda task retries on Lambda infrastructure errors specifically. The backoff intervals are conservative (2s base for Lambda/DynamoDB, 5s for Polly) with a 2x rate. This covers transient infrastructure faults but does not cover application-level exceptions thrown by the handler itself.

The error catch strategy was narrowed from catch-all (`States.ALL` default) to explicit error lists on ProcessAudio and PollyTask. This introduces a gap: errors not in the enumerated list (like the handler's `ValueError` or an unexpected `RuntimeError`) will not be caught and will cause the state machine execution to fail outright without updating the DynamoDB record to FAILED status.

The CloudWatch Alarms are created without `alarm_actions`, so they will transition to ALARM state in CloudWatch but not notify anyone. The `requestId` injection into the Lambda response payload provides downstream correlation, but the tests verify the response shape without capturing actual log output.

1. **Uncaught application errors in ProcessAudio** — The catch block on ProcessAudio only lists Lambda infrastructure errors and `States.TaskFailed`. 4. **Alarm actions not wired** — Both CloudWatch Alarms are created without `alarm_actions`, so they transition state but don't notify anyone. Wire them to the existing SNS failure topic or an ops topic.

## Catch specificity gap on ProcessAudio

The catch block on `process_audio_task` enumerates four error types:

```python
process_audio_task.add_catch(
    update_status_failed,
    errors=[
        "Lambda.ServiceException",
        "Lambda.AWSLambdaException",
        "Lambda.SdkClientException",
        "States.TaskFailed",
    ],
    result_path="$.error",
)
```

These cover Lambda infrastructure failures but not application-level exceptions. When the handler raises `ValueError("Missing required fields: object.key or bucket.name")`, Step Functions surfaces this as an unhandled error not in this list. The execution will fail without routing through UpdateStatusFailed, leaving the DynamoDB record stuck in PROCESSING status permanently.

The pre-existing code used a catch-all (`add_catch(update_status_failed, result_path="$.error")` with no `errors` parameter, defaulting to `States.ALL`). By narrowing to specific errors, the change gains precision but loses coverage of the handler's own exceptions. A safe fix is to add a second catch for `States.ALL` as a fallback after the specific one.

## PublishFailedNotification message assumes a path that doesn't exist on error

The `PublishFailedNotification` task constructs its message from `$.processAudioResult.Payload.validationError`:

```python
"reason": sfn.JsonPath.string_at(
    "$.processAudioResult.Payload.validationError"
),
```

This path is populated only when ProcessAudio completes successfully but the audio is invalid (the "otherwise" branch of ValidateInput). When the error catch fires, the state context contains `$.error.Error` and `$.error.Cause` but no `$.processAudioResult`. The notification task will fail trying to resolve the path, which is caught by its own `add_catch(fail_state)` — so the execution reaches Fail, but no failure notification is actually sent to SNS. This is a pre-existing design issue that becomes more impactful now that the error routing is more explicit and more paths lead through this notification task.

## PollyTask error surface area

Both retry and catch on `polly_task` target only `States.TaskFailed`. The Polly `startSpeechSynthesisTask` API can return throttling exceptions or limit-exceeded errors that Step Functions may classify differently. If Polly's error classification doesn't match `States.TaskFailed`, the error will escape both retry and catch, causing an unhandled execution failure. Adding `States.ALL` as a fallback catch (routing to UpdateStatusFailed) would prevent silent failures.

## CloudWatch Alarms without actions

```python
state_machine.metric_failed(...).create_alarm(
    self,
    "StateMachineExecutionFailuresAlarm",
    alarm_description="Alarm when state machine executions fail",
    threshold=1,
    evaluation_periods=1,
    comparison_operator=cloudwatch.ComparisonOperator.GREATER_THAN_OR_EQUAL_TO_THRESHOLD,
)
```

Without `actions` or `alarm_actions`, the alarms transition state in CloudWatch but don't trigger any notification. The stack already has an SNS `failed_topic` that could be wired as an alarm action to close the loop.

## Test coverage gaps

The state machine definition tests use substring matching against the serialized JSON (`'"IntervalSeconds":2'` etc.). A CDK version update that changes field ordering or JSON serialization will break these assertions without any behavioral change. The X-Ray and CloudWatch Alarm tests use `has_resource_properties`, which is more resilient.

The structured logging test invokes the handler and asserts `requestId` appears in the response dict, but does not capture `logger.info`/`logger.error` calls. The test verifies response shape, not actual logging behavior. The logging contract (JSON format, status field, requestId correlation) is untested.
