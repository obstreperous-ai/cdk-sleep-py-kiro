# SleepAudioProcessor Lambda skeleton and state machine integration

The change adds a Lambda function (`SleepAudioProcessor`) that sits between the DynamoDB initial write and the Polly task in the Step Functions pipeline. The Lambda receives the full state machine input (S3 bucket/object details), logs it, validates that required fields are present, and returns enriched metadata. The CDK stack wires it in with `LambdaInvoke`, grants it DynamoDB read/write access, and chains error handling to the existing `UpdateStatusFailed` path.

Watch for: The Lambda receives DynamoDB grants but never actually uses them at runtime (confirmed). The `LambdaInvoke` uses default response wrapping (`payload_response_only=False`), nesting the handler's return under `$.processAudioResult.Payload` — future consumers need to account for this depth (confirmed). No unit tests exist for the Lambda handler code itself (confirmed).

## High-level view

The Lambda handler is a placeholder that validates input shape and returns a static metadata envelope. It currently doesn't use the DynamoDB access it's been granted — the grant is forward-looking for when the handler starts enriching records, but it inflates the IAM attack surface of an otherwise no-op function.

The handler expects `event.object.key` and `event.bucket.name` at the top level. By the time the Lambda is invoked, the state carries additional keys (`dynamoResult`) from the preceding step. The handler's `.get()` pattern is safe against extras today, but future steps that overwrite `$.object` or `$.bucket` would cause silent validation failures. Scoping the `LambdaInvoke` payload to only the needed fields would prevent this class of bug.

Test coverage validates CDK infrastructure properties thoroughly (resource existence, IAM grants, state machine topology, error handling) but does not include any unit tests for the Lambda handler function itself — input validation logic and error paths are exercised only at deploy time.

## DynamoDB grant without current usage

The Lambda receives `grant_read_write_data` (10 DynamoDB actions including `PutItem`, `UpdateItem`, `DeleteItem`, `BatchWriteItem`) but performs no DynamoDB calls. The `TABLE_NAME` environment variable is read but only echoed back in the response. If the Lambda is compromised, the attacker has write access to the metadata table. In a production scenario, the grant should be added at the same commit that adds the DynamoDB calls.

## Input payload shape and `LambdaInvoke` defaults

The synthesized state machine uses `"Payload.$": "$"`, which passes the entire state object to the Lambda. After `WriteInitialRecord`, the state looks like:

```json
{
  "bucket": { "name": "..." },
  "object": { "key": "..." },
  "dynamoResult": { ... }
}
```

The handler's `event.get("object", {}).get("key", "")` correctly extracts the needed fields and ignores extras. However, `LambdaInvoke` with default `payload_response_only=False` wraps the Lambda return value in `{ "Payload": {...}, "SdkHttpMetadata": {...}, ... }` under `$.processAudioResult`. Downstream steps must access the Lambda's return via `$.processAudioResult.Payload.audioId`. Specifying `payload_response_only=True` on the `LambdaInvoke` would flatten this to just the handler's return dict under `$.processAudioResult`, simplifying downstream access when consumers are added.

## Missing handler unit tests

The 10 CDK assertion tests validate infrastructure shape. None test the Lambda handler's Python logic. The handler has two behavioral paths worth covering:

- Valid input returns the expected metadata envelope with `processorStatus: "PROCESSED"`.
- Missing `object.key` or `bucket.name` raises `ValueError`.

Without handler tests, a regression in input parsing won't be caught until integration testing or production.
