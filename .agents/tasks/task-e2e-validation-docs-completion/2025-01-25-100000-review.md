# End-to-end validation tests and documentation overhaul

This change adds 116 new end-to-end tests across two files (`test_e2e_flow.py` and additions to `test_pipeline_e2e.py`) that validate the Lambda handler's complete processing paths and the Step Functions state machine's DynamoDB/SNS integration contracts. Alongside the tests, the entire documentation surface is rewritten: README.md is replaced from CDK boilerplate to project-specific docs, ARCHITECTURE.md gains a quick-reference table and testing strategy section, AGENT_GUIDELINES.md gets troubleshooting and dependency sections, and a new SUMMARY.md captures key decisions.

**Watch for:** The new e2e tests are entirely mock-based with no integration-level verification, meaning contract drift between the Lambda and the state machine definitions would go undetected (likely). The `_parse_definition` / `_get_definition` helper is duplicated across both test files rather than shared via conftest (confirmed). Documentation claims "298 tests" which will drift as the codebase evolves (possible — minor).

## High-level view

The test additions split into two categories: `test_e2e_flow.py` exercises the Lambda handler in isolation (happy path audio, happy path TTS, DynamoDB failure tracking, input validation, and parameterized format coverage), while the `test_pipeline_e2e.py` additions validate the Step Functions state machine definition for retry configuration, SNS message payloads, and DynamoDB field contracts. Both use the same pattern of parsing the `Fn::Join`-serialized definition string, and both duplicate the parsing helper rather than extracting it to a shared fixture.

The README replaces generic CDK boilerplate entirely. ARCHITECTURE.md lists the Polly Task as "Implemented" in the Quick Reference table, though SUMMARY.md clarifies it's a placeholder for future advanced workflows — the table doesn't surface this nuance, which could mislead someone scanning only the quick reference.

1. **Duplicated definition parser** — `_get_definition` in `test_e2e_flow.py` and `_parse_definition` in `test_pipeline_e2e.py` are identical functions. Extract to `conftest.py` or a shared helper to eliminate the maintenance burden of keeping two copies in sync.

A single assertion or schema object shared between both test suites would catch drift.
3. **Hardcoded test count in docs** — SUMMARY.md and other docs reference "298 tests". This number will become stale the next time tests are added or removed. Consider removing it or generating it.
4. **Test granularity vs. parametrize** — Many test classes in `test_pipeline_e2e.py` (e.g., `TestRetryConfiguration`) contain 6-8 near-identical tests that differ only by which state and which retry property is being checked. A single parametrized test would be more maintainable and would make it obvious when one state has different retry semantics from another.

## Duplicated state machine parsing logic

Both `test_e2e_flow.py` and `test_pipeline_e2e.py` contain an identical helper that extracts the state machine definition from the synthesized CloudFormation template:

```python
def _get_definition(template):
    sm_resources = template.find_resources("AWS::StepFunctions::StateMachine")
    resource = list(sm_resources.values())[0]
    definition_str = resource["Properties"]["DefinitionString"]
    if isinstance(definition_str, dict) and "Fn::Join" in definition_str:
        parts = definition_str["Fn::Join"][1]
        joined = "".join(
            p if isinstance(p, str) else "PLACEHOLDER" for p in parts
        )
        return json.loads(joined)
    return json.loads(definition_str) if isinstance(definition_str, str) else definition_str
```

Having two identical implementations means a bug fix or format change to the definition serialization requires updating both files. Moving this to `conftest.py` as a fixture (e.g., `@pytest.fixture def state_machine_definition(template)`) would also simplify each test method since it wouldn't need to call the helper explicitly.

## Lambda-to-state-machine contract gap

The Lambda handler returns a dict with keys like `outputKey`, `outputBucket`, `fileSize`, `status`, and `valid`. The state machine definition references these via JSONPath expressions such as `$.processAudioResult.Payload.outputKey`. The test suites validate each side independently:

- `TestHappyPathAudioProcessing` asserts `result["outputKey"].startswith("processed/")`
- `TestDynamoDBMetadataLifecycle` asserts `expr_values[":outputKey"] == {"S.$": "$.processAudioResult.Payload.outputKey"}`

If the Lambda changed its return key from `outputKey` to `output_key`, both test suites would still pass but the deployed pipeline would break. A shared constant or a single integration test that verifies the Lambda's return schema matches the state machine's expected input paths would close this gap.

## Test granularity in retry and notification sections

`TestRetryConfiguration` has separate test methods for `IntervalSeconds`, `MaxAttempts`, and `BackoffRate` on both `ProcessAudio` and `PollyTask` — six tests that each parse the full definition, navigate to the same state, and check one integer. If a third state gains retry configuration, you'd write three more near-identical methods. Parametrizing over `(state_name, retry_field, expected_value)` tuples would collapse this to one test with six parameter sets while preserving the same coverage.

The same pattern appears in `TestNotificationContent` (8 tests checking individual fields of the SNS message payload) — the signal-to-noise ratio in test output drops when you have eight green dots that all assert different keys in the same dict.

## Polly Task status ambiguity in Quick Reference

ARCHITECTURE.md's Quick Reference table lists "Polly Task (placeholder)" with status "Implemented". SUMMARY.md explains this is a Step Functions `CallAwsService` state that exists as a placeholder for future advanced workflows, with actual Polly synthesis handled directly by the Lambda. A reader scanning only the Quick Reference table would reasonably conclude Polly integration is fully operational via the Step Functions SDK integration path, which isn't the case. Adding "(placeholder)" to the Status column or a footnote would resolve this.
