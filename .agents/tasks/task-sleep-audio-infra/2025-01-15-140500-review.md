# S3 buckets and EventBridge rule for sleep audio pipeline (v2)

The change adds two S3 buckets (input and output) and an EventBridge rule that triggers on Object Created events from the input bucket, with a CloudWatch Log Group placeholder target. Both buckets are encrypted, versioned, block public access, and have `auto_delete_objects=True` for clean teardown. The test suite uses `find_resources` for count-based assertions and asserts the EventBridge rule's `detail.bucket.name` structure. All four issues from the v1 review have been addressed.

**Watch for:** The CDK-internal Lambda log group (created by `auto_delete_objects`) retains logs for 731 days with no explicit override — a minor cost concern in long-lived dev stacks (likely). The EventBridge rule test asserts the `detail.bucket.name` path exists but still uses `Match.any_value()` on the name array itself, so it cannot distinguish input-bucket wiring from output-bucket wiring (confirmed).

---

## High-level view

The bucket tests now use `find_resources` and assert exactly two matches, which is a meaningful improvement over the v1 approach — the test would fail if either bucket diverged from the shared security contract. The input bucket is differentiated by a `Custom::S3BucketNotifications` existence and count check, which indirectly confirms it's the one with EventBridge enabled.

The EventBridge rule test moved from `Match.any_value()` on the entire `detail` block to asserting the nested `detail.bucket.name` key structure. This catches category errors (wrong detail shape) but still accepts any value in the `name` array, so a wiring mistake (e.g., pointing at the output bucket) would not be caught. This is a residual gap but substantially less risky than v1.

The `auto_delete_objects=True` addition creates a CDK-managed custom resource Lambda with its own log group. That internal log group defaults to 731-day retention and `Retain` deletion policy — it will persist after stack destroy and accumulate cost in long-lived accounts. This is CDK's standard behavior and not a blocking issue, but worth noting.

## EventBridge rule test: structural vs. wiring verification

The synthesized template shows:

```json
"detail": {
  "bucket": {
    "name": [{ "Ref": "SleepAudioInputBucket1923DAE4" }]
  }
}
```

The test asserts:

```python
"detail": {
    "bucket": {
        "name": Match.any_value()
    }
}
```

This confirms the rule filters on `detail.bucket.name` (structural correctness) but doesn't verify the *value* resolves to the input bucket. A refactoring that accidentally wired the output bucket into the rule's detail pattern would pass this test. Asserting the `Ref` target would close this gap:

```python
"name": Match.array_with([
    Match.object_like({"Ref": Match.string_like_regexp("SleepAudioInputBucket")})
])
```

This isn't blocking — the notification-count test (`resource_count_is("Custom::S3BucketNotifications", 1)`) provides indirect evidence of correct wiring — but it leaves a coverage gap in the rule itself.

## CDK-internal log group lifecycle

`auto_delete_objects=True` injects a custom resource Lambda (`AWS679f53fac002430cb0da5b7982bd2287`). CDK creates a companion log group for that Lambda with:

- `RetentionInDays: 731` (CDK default, not overridable per-bucket)
- `DeletionPolicy: Retain` (survives stack destroy)

In a dev account that frequently creates and destroys stacks, these orphaned log groups accumulate. The CDK team considers this acceptable behavior. Setting the `@aws-cdk/custom-resources:logRetentionDays` context key can control this globally, or it can simply be accepted as a known trade-off for dev stacks.
