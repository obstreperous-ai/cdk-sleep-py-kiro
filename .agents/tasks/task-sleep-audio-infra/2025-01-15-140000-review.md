# S3 buckets and EventBridge rule for sleep audio pipeline

The change adds two S3 buckets (input and output) and an EventBridge rule that triggers on Object Created events from the input bucket, implemented via CDK L2 constructs with a TDD approach. Both buckets are encrypted (SSE-S3), versioned, and block all public access. The EventBridge rule routes to a placeholder CloudWatch Log Group target. ARCHITECTURE.md is updated with an implementation status table and a visual legend in the Mermaid diagram.

**Watch for:** Tests cannot distinguish between the two buckets since they assert identical properties on `AWS::S3::Bucket` without identifying which bucket is which (confirmed). The EventBridge rule's `detail.bucket.name` filter uses a CloudFormation `Ref` token, which resolves correctly at deploy time but the test's `assertions.Match.any_value()` on the `detail` field doesn't actually verify the bucket-name linkage (confirmed). The log group target uses CDK's default 731-day retention with no explicit override, which may not match the project's per-environment retention strategy described in ARCHITECTURE.md (likely).

---

## High-level view

Both tests (`test_input_bucket_has_correct_properties` and `test_output_bucket_has_correct_properties`) assert the same property bag against `AWS::S3::Bucket`, so either test passes if *any* bucket in the template matches. If the output bucket accidentally dropped versioning, the test would still pass because the input bucket satisfies the assertion. The tests prove "at least one bucket has these properties" not "each specific bucket has its properties."

The CDK token for `input_bucket.bucket_name` synthesizes to a `Ref` intrinsic, so EventBridge will match the physical bucket name at deploy time. The test uses `assertions.Match.any_value()` on the entire `detail` block, so it doesn't verify that the rule is actually wired to the *input* bucket rather than some hardcoded string or the wrong bucket.

The 731-day retention on the placeholder log group (CDK default when no `retention` kwarg is passed) will silently accumulate logs at cost if not revisited before production, contradicting the 7/30/90-day strategy in ARCHITECTURE.md.

## Test specificity for dual-bucket stacks

The two bucket tests:

```python
def test_input_bucket_has_correct_properties(template):
    template.has_resource_properties("AWS::S3::Bucket", { ... })

def test_output_bucket_has_correct_properties(template):
    template.has_resource_properties("AWS::S3::Bucket", { ... })
```

assert *identical* property sets. `has_resource_properties` succeeds if *any* resource of that type matches. The test suite would pass with only a single bucket in the stack, and would not catch a regression where the output bucket loses encryption or versioning while the input bucket retains it.

The input bucket has `event_bridge_enabled=True`, which synthesizes to a `Custom::S3BucketNotifications` resource referencing that bucket's ARN. A stronger approach: assert the notification custom resource's `BucketName` references the same logical ID as the bucket with EventBridge-specific properties, or use `template.find_resources("AWS::S3::Bucket", {...})` and confirm exactly *two* distinct matches.

## EventBridge rule wiring gap in tests

The synthesized template shows:

```json
"EventPattern": {
  "detail": { "bucket": { "name": [{"Ref": "SleepAudioInputBucket1923DAE4"}] } },
  "detail-type": ["Object Created"],
  "source": ["aws.s3"]
}
```

The test:

```python
template.has_resource_properties("AWS::Events::Rule", {
    "EventPattern": {
        "source": ["aws.s3"],
        "detail-type": ["Object Created"],
        "detail": assertions.Match.any_value(),
    },
})
```

This only verifies the rule has *some* detail blockâ€”it could be filtering on a completely wrong field and still pass. To lock down the contract, assert against the `Ref` token structure or at minimum assert `detail.bucket.name` is a non-empty array.

## Stack teardown with non-empty DESTROY buckets

Both buckets set `removal_policy=RemovalPolicy.DESTROY` without `auto_delete_objects=True`. S3 refuses to delete a non-empty bucket, so `cdk destroy` will fail after any manual uploads. Adding `auto_delete_objects=True` instructs CDK to create a custom resource that empties the bucket before deletionâ€”safe for dev stacks and expected when `DESTROY` is chosen.

## Log group retention and the environment model

ARCHITECTURE.md defines a per-environment retention strategy (7/30/90 days). The `AudioUploadRuleLogGroup` already has a concrete resource with 731-day retention baked in. Worth adding an explicit `retention=logs.RetentionDays.ONE_WEEK` now so it doesn't drift into production at two years of retention.
