# Multi-environment support and CDK Pipeline skeleton

This change introduces environment-aware resource configuration to the CDK stack and a self-mutating CDK Pipeline construct for CI/CD. The `environment` CDK context key (dev/stage/prod) now governs log retention, removal policies, and S3 auto-delete behavior across all resources. A new `PipelineStack` wires GitHub source through CodePipeline to deploy the application stack. The diff also adds 28 new tests covering environment differentiation, pipeline synthesis, error path routing, and payload scoping.

**Watch for:** The pipeline uses a hardcoded placeholder CodeStar connection ARN and a generic `owner/repo` source identifier — deploying the PipelineStack without replacing these will fail at runtime (confirmed). The `ApplicationStage` inside the pipeline doesn't pass environment context to `CdkBaseStack`, so the pipeline always deploys with dev defaults regardless of the stage name (confirmed). The CI workflow runs `cdk synth` four times (bare + three environments) which is redundant since bare synth already uses dev defaults (confirmed, minor).

## High-level view

The environment configuration uses a module-level `ENV_CONFIG` dict keyed by environment name, read at stack construction via `try_get_context`. The fallback-to-dev pattern is applied consistently in both `app.py` and `CdkBaseStack`. However, the pipeline construct doesn't participate in this system — `ApplicationStage` instantiates `CdkBaseStack` without forwarding any environment context, creating a gap between the architecture documentation (which describes multi-stage dev → stage → prod deployment) and what the pipeline actually produces.

The `PipelineStack` is a minimal skeleton: single-stage deployment with placeholder credentials. It synthesizes and has tests, but cannot function as-is. The architecture docs describe future multi-stage deployment (with approval gates) that doesn't exist in code yet — anyone reading ARCHITECTURE.md expecting a working multi-stage pipeline will be misled.

The multi-environment tests verify CloudFormation-level output (DeletionPolicy, RetentionInDays) rather than CDK abstractions. The error path routing tests parse the CDK-generated state machine definition (handling `Fn::Join` token replacement). Neither test suite covers the `deploy_pipeline` conditional gating in `app.py` or the `ApplicationStage` → `CdkBaseStack` wiring within the pipeline.

## ApplicationStage doesn't propagate environment context

The pipeline's `ApplicationStage` hard-codes a bare `CdkBaseStack(self, "CdkBaseStack")` call on line 19 of `pipeline_stack.py`. Since no environment context is set at the stage level, `self.node.try_get_context("environment")` returns `None` inside `CdkBaseStack`, which falls through to `"dev"` defaults. The ARCHITECTURE.md describes a `Deploy Dev → Deploy Stage → Deploy Prod` flow, but the current pipeline would deploy three identical dev-configured stacks.

To fix: accept an `environment` parameter in `ApplicationStage.__init__`, store it, and either set it as context on the stage node or pass it as a stack property. Then call `pipeline.add_stage(...)` multiple times with different environments.

## Placeholder credentials in PipelineStack

```python
connection_arn="arn:aws:codestar-connections:us-east-1:123456789012:connection/placeholder",
```

The stack synthesizes successfully while being non-functional. Deploying this stack creates a real CodePipeline resource that immediately fails on the Source action. There's no runtime guard. Consider either:
- Reading the connection ARN from context with a descriptive error when missing
- Adding a comment-level `# TODO` that grep-based CI checks can catch before deploy

## Silent fallback on invalid environment values

```python
config = ENV_CONFIG.get(environment, ENV_CONFIG["dev"])
```

If someone passes `-c environment=staging` (note: "staging" not "stage"), the stack silently uses dev configuration. In production workflows this could lead to deploying DESTROY removal policies to what the operator believes is a production stack. A validation step that checks `environment in ENV_CONFIG` and raises a clear error would prevent this class of misconfiguration.

## Stack ID change is a breaking change for existing deployments

The stack ID changed from `"CdkBaseStack"` to `f"CdkBaseStack-{environment}"`. Any existing deployment using the old stack name will see CloudFormation attempt to create a new stack rather than update the existing one. The old resources (S3 buckets, DynamoDB table, state machine) remain orphaned under the old stack. This is fine for greenfield deployment but warrants a migration note for anyone with an existing `CdkBaseStack` in their account.

## Test coverage gaps

The pipeline construct tests verify synthesis and stage presence but don't test the `deploy_pipeline` conditional gating in `app.py` — there's no test that verifies `PipelineStack` is absent when context is not `"true"` or present when it is.

Not tested: the conditional `deploy_pipeline` context gating in `app.py`, invalid environment context values, and the `ApplicationStage` → `CdkBaseStack` wiring within the pipeline (whether the deployed stack has the correct resources).
