# Multi-environment support and CDK Pipeline skeleton (v2)

This change adds environment-differentiated resource configuration (log retention, removal policies, auto-delete behavior) controlled by CDK context, a self-mutating CDK Pipeline skeleton, expanded test coverage for error path routing and payload scoping, and updated documentation. The v1 review identified four issues: ApplicationStage ignoring environment context, a placeholder connection ARN, unvalidated environment values, and a redundant CI synth step. Three were fixed; the placeholder ARN was acknowledged as intentional with a TODO comment.

**Watch for:** The CI `cdk diff` step still runs without environment context and without AWS credentials — it always passes vacuously and provides no signal (confirmed, low severity). The pipeline deploys only a single `dev` stage despite ARCHITECTURE.md describing a three-stage flow (confirmed, documentation gap).

**Verdict**: APPROVED

## High-level view

The environment system uses a module-level `ENV_CONFIG` dict with validation that rejects unknown values at construction time. The fix for ApplicationStage uses `node.set_context` to propagate environment into the stage's child stack, which is the correct CDK Pipelines pattern.

The pipeline remains a single-stage skeleton. ARCHITECTURE.md shows a three-stage pipeline diagram (dev → stage → prod with yellow "planned" markers on stage/prod) but the code only calls `add_stage` once with `environment="dev"`. The diagram legend clarifies this distinction, though the visual weight of the three-stage flow may mislead readers.

The CI workflow replaced the bare `cdk synth` with explicit per-environment synth steps. The residual `cdk diff` step runs without context and without AWS credentials, so it always passes vacuously via `continue-on-error: true`.

## CI diff step provides no signal

The `cdk diff` step in `ci.yml` runs without `-c environment=...`:

```yaml
- name: CDK diff
  run: cdk diff
  continue-on-error: true
```

Without context, `app.py` defaults to `"dev"` and produces stack `CdkBaseStack-dev`. Without AWS credentials, the diff command cannot contact CloudFormation. Combined with `continue-on-error: true`, this step always passes vacuously. It should either be removed, gated behind a credential-availability condition, or moved to a deployment workflow that has AWS access.

## Single-stage pipeline vs. multi-stage documentation

The `PipelineStack` calls `add_stage` once:

```python
pipeline.add_stage(ApplicationStage(self, "Deploy", environment="dev"))
```

ARCHITECTURE.md shows `DeployDev → DeployStage → DeployProd` with stage/prod nodes styled yellow and a legend marking them "planned." The visual emphasis on a full three-stage flow may mislead readers into expecting multi-environment deployment from this pipeline. The `ApplicationStage` now correctly propagates environment context via `node.set_context`, so adding stages is straightforward when ready.
